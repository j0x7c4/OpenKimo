# Contributing to OpenKimo

Thank you for your interest in contributing!

## How to Contribute

1. **Fork** the repository and create your branch from `main`.
2. **Make your changes** — bug fixes, features, docs, or tests are all welcome.
3. **Test your changes** locally with `docker-compose up -d --build`.
4. **Submit a Pull Request** with a clear description of the changes.

## Development Environment Setup

The fastest way to get a dev environment is with Docker Compose:

```bash
cp .env.example .env
# Edit .env and add your LLM API keys
docker compose -f docker-compose.yml up --build
```

This builds both the `gateway` and `sandbox` images and mounts the source code with live-reload where applicable.

If you prefer to run Python natively for faster iteration on the gateway:

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements-dev.txt
uvicorn app.main:app --reload --port 8000
```

> **Note:** The sandbox container still requires Docker to be running on your host.

## Code Style

### Python (Gateway / Sandboxes)
- We use **[Ruff](https://docs.astral.sh/ruff/)** for linting and formatting.
- Run before committing:
  ```bash
  ruff check .
  ruff format .
  ```
- Target Python version: **3.11+**.
- Follow PEP 8 with the exceptions configured in `pyproject.toml`.

### TypeScript / JavaScript (Web UI / Browser extensions)
- Use the project's own `eslint` and `prettier` configs (found in `package.json`).
- Run:
  ```bash
  npm run lint
  npm run format
  ```

### General
- Keep functions small and focused.
- Add docstrings for public APIs and complex logic.
- Write comments in English.

## Commit Message Convention

We follow [Conventional Commits](https://www.conventionalcommits.org/) to generate the changelog automatically.

```
<type>(<scope>): <description>

[optional body]

[optional footer(s)]
```

Common types:
- `feat`: New feature
- `fix`: Bug fix
- `docs`: Documentation only
- `style`: Code style (formatting, missing semicolons, etc.)
- `refactor`: Code change that neither fixes a bug nor adds a feature
- `test`: Adding or correcting tests
- `chore`: Build process, dependencies, tooling

Examples:
```
feat(sandbox): add support for custom Docker network modes
fix(gateway): handle Anthropic rate-limit headers correctly
docs: update README with podman compatibility notes
```

## AI-Assisted PRs Are Welcome!

We use AI tools ourselves and welcome contributions that are partially or fully generated by AI. **Just mark it in the PR description** so reviewers know what to expect. A simple line like the following is enough:

> This PR was AI-assisted (Claude / ChatGPT / Copilot / etc.).

If you used AI for code generation, please still:
- Read and understand the code before submitting.
- Verify that tests pass and the change behaves as intended.
- Ensure licensing and attribution are correct for any copied snippets.

## Testing

- All bug fixes must include a test that would have caught the bug.
- New features must include tests covering the happy path and at least one error case.
- Run the test suite before opening a PR:
  ```bash
  pytest tests/ -q
  ```
- If your change touches the sandbox, verify it manually with:
  ```bash
  docker compose up --build
  # Then run a session that exercises your change
  ```

## Pull Request Checklist

- [ ] Code follows the style guidelines (Ruff / ESLint / Prettier).
- [ ] Tests pass locally.
- [ ] Commit messages follow Conventional Commits.
- [ ] PR description explains *what* changed and *why*.
- [ ] If AI-assisted, noted in the PR description.
- [ ] Documentation updated if user-facing behavior changed.

## Code of Conduct

We are committed to providing a friendly, safe, and welcoming environment for all contributors.

- Be respectful and constructive in all interactions.
- Welcome newcomers and help them learn the codebase.
- Disagree with ideas, not people.
- Harassment or discriminatory behavior of any kind will not be tolerated.

If you experience or witness unacceptable behavior, please contact the maintainers at `conduct@openkimo.dev`.
